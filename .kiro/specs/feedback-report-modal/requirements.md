# Requirements Document

## Introduction

SoulReel currently has no in-app mechanism for users to report bugs or suggest features. Users who encounter issues or have ideas must resort to external channels (email, social media), which creates friction and means valuable feedback is lost. This feature adds a "Report a Bug or Suggest a Feature" menu item to the existing UserMenu dropdown, opening a modal dialog with a structured feedback form. The form supports both bug reports and feature requests, pre-fills known user information from AuthContext, and submits to a new backend Lambda endpoint. Submissions are stored in a DynamoDB table and automatically classified by an LLM (Claude Haiku via Bedrock) as "bug" or "feature request", with a one-sentence AI-generated summary. A new "Bugs and Feature Requests" section in the admin console provides a sortable table for reviewing, archiving, and deleting reports, with a detail view for each submission. No email notifications are sent. The experience is designed to feel warm, encouraging, and natural — consistent with SoulReel's empathetic tone — so users feel invited to share without feeling burdened. The entire flow uses existing shadcn/ui components and follows established patterns for state management, validation, loading states, and error handling.

## Glossary

- **Feedback_Modal**: The shadcn/ui Dialog component that contains the feedback submission form, opened from the UserMenu dropdown
- **Feedback_Form**: The React form component rendered inside the Feedback_Modal, containing report type selection, subject, description, optional file attachment metadata, and pre-filled user information
- **Feedback_Menu_Item**: The new DropdownMenuItem in UserMenu.tsx labeled "Report a Bug or Suggest a Feature", using a lucide-react icon (Bug or Lightbulb)
- **Feedback_Submission_Endpoint**: The POST /feedback API Gateway endpoint (Cognito-authenticated) that accepts structured feedback reports from the frontend, stores them in the Feedback_Reports_Table, and triggers AI classification
- **Feedback_Ingestion_Lambda**: The AWS Lambda function backing the Feedback_Submission_Endpoint, responsible for validating, storing, and classifying feedback reports
- **Feedback_Reports_Table**: The DynamoDB table that stores all submitted feedback reports, including user-provided fields, AI classification, AI summary, and status (active/archived)
- **Feedback_Admin_Endpoint**: The set of API Gateway endpoints (Cognito-authenticated, admin-only) for listing, retrieving, archiving, and deleting feedback reports
- **Feedback_Admin_Lambda**: The AWS Lambda function backing the Feedback_Admin_Endpoint, responsible for querying, updating, and deleting records in the Feedback_Reports_Table
- **AI_Classification**: The automatic classification of a feedback submission as "bug" or "feature_request" by Claude Haiku via Amazon Bedrock, performed during ingestion
- **AI_Summary**: A one-sentence summary of the feedback report generated by Claude Haiku via Amazon Bedrock, stored alongside the report
- **Admin_Feedback_Page**: The new page in the admin console under the "FEEDBACK" navigation section, displaying a sortable table of all feedback reports
- **Report_Type**: An enumerated value indicating whether the feedback is a "bug" report or a "feature" request, as selected by the user
- **Report_Status**: An enumerated value indicating whether a feedback report is "active" or "archived"
- **Thank_You_Message**: The confirmation message displayed inside the Feedback_Modal after successful submission, shown briefly before the modal auto-closes
- **UserMenu**: The existing dropdown menu component (UserMenu.tsx) that provides access to user profile actions, settings, and navigation

## Requirements

### Requirement 1: Feedback Menu Item in UserMenu

**User Story:** As a user, I want to see a clearly visible menu item in the UserMenu dropdown for reporting bugs or suggesting features, so that I can easily find and access the feedback form without leaving my current workflow.

#### Acceptance Criteria

1. THE UserMenu SHALL display a Feedback_Menu_Item labeled "Report a Bug or Suggest a Feature" using a lucide-react icon (Bug or Lightbulb) with the same styling pattern as other menu items (min-h-[44px], text-sm text-legacy-navy, legacy-purple icon, hover:bg-legacy-purple/10)
2. THE Feedback_Menu_Item SHALL be placed in the navigation section of the UserMenu dropdown, between the "Your Data" item and the disabled "Settings" item, grouped logically with privacy-related items
3. THE Feedback_Menu_Item SHALL be visible to all authenticated users regardless of persona type (legacy_maker, legacy_benefactor)
4. WHEN the user clicks the Feedback_Menu_Item, THE UserMenu SHALL open the Feedback_Modal dialog
5. WHEN the user clicks the Feedback_Menu_Item, THE UserMenu dropdown SHALL close before the Feedback_Modal opens, consistent with how other dialog-triggering menu items behave (Edit Profile, Change Password, Security & Privacy)

### Requirement 2: Feedback Modal Dialog

**User Story:** As a user, I want the feedback form to appear in a modal dialog that is large enough for comfortable writing, so that I can compose detailed reports without feeling cramped.

#### Acceptance Criteria

1. THE Feedback_Modal SHALL use the existing shadcn/ui Dialog component (Dialog, DialogContent, DialogHeader, DialogTitle, DialogDescription, DialogFooter) for visual consistency with ProfileDialog and PasswordDialog
2. THE Feedback_Modal SHALL render at a width of at least 540px on desktop viewports (sm breakpoint and above) to provide comfortable writing space
3. THE Feedback_Modal SHALL be fully responsive, filling the available viewport width on mobile devices with appropriate padding
4. THE Feedback_Modal SHALL include a DialogTitle of "Report a Bug or Suggest a Feature" and a DialogDescription with warm, encouraging copy (e.g., "We appreciate you taking the time to help us improve SoulReel.")
5. THE Feedback_Modal SHALL include a visible close button and support closing via the Escape key, consistent with other dialogs in the application
6. WHEN the Feedback_Modal opens, THE Feedback_Form SHALL reset all fields to their default state (empty subject, empty description, default report type) and pre-fill user email and name from AuthContext
7. THE Feedback_Modal SHALL trap keyboard focus within the dialog while open, following WAI-ARIA dialog accessibility requirements

### Requirement 3: Feedback Form Structure and Validation

**User Story:** As a user, I want a structured form that guides me through providing the right information for a bug report or feature request, so that my feedback is actionable for the development team.

#### Acceptance Criteria

1. THE Feedback_Form SHALL include a Report_Type selector (using shadcn/ui Select) with two options: "Bug Report" and "Feature Request", defaulting to no selection so the user makes an explicit choice
2. THE Feedback_Form SHALL include a subject field (text input, required, maximum 200 characters) for a brief summary of the bug or feature
3. THE Feedback_Form SHALL include a description field (Textarea, required, minimum 10 characters, maximum 5000 characters) for detailed information about the bug or feature request
4. THE Feedback_Form SHALL display the user's pre-filled email address (from AuthContext) as a read-only field so the user knows which account the report is associated with
5. THE Feedback_Form SHALL display the user's pre-filled name (firstName and lastName from AuthContext) as a read-only field, falling back to "Anonymous" when name fields are not available
6. WHEN the user attempts to submit the form with missing required fields (Report_Type, subject, or description), THE Feedback_Form SHALL display inline validation errors below each invalid field without submitting the request
7. WHEN the user attempts to submit a description shorter than 10 characters, THE Feedback_Form SHALL display a validation error indicating the minimum length requirement
8. THE Feedback_Form SHALL include a submit button labeled "Submit Feedback" in the DialogFooter, styled consistently with other dialog submit buttons in the application

### Requirement 4: Form Submission and Loading States

**User Story:** As a user, I want clear feedback during and after form submission, so that I know my report was received and I feel confident it will be reviewed.

#### Acceptance Criteria

1. WHEN the user clicks the submit button with valid form data, THE Feedback_Form SHALL send a POST request to the Feedback_Submission_Endpoint with the report type, subject, description, user email, and user name
2. WHILE the Feedback_Form is submitting, THE submit button SHALL display a Loader2 spinner icon and the text "Submitting..." and SHALL be disabled to prevent duplicate submissions
3. WHILE the Feedback_Form is submitting, THE Feedback_Modal close button and Escape key SHALL be disabled to prevent the user from dismissing the modal during submission
4. WHEN the Feedback_Submission_Endpoint returns a successful response (HTTP 200), THE Feedback_Modal SHALL display the Thank_You_Message in place of the form
5. THE Thank_You_Message SHALL display warm, appreciative copy (e.g., "Thank you for your feedback. We read every report and it helps us make SoulReel better for everyone.") with a checkmark or heart icon
6. WHEN the Thank_You_Message is displayed, THE Feedback_Modal SHALL automatically close after a 2.5-second delay
7. IF the Feedback_Submission_Endpoint returns an error response or the request fails due to a network error, THEN THE Feedback_Form SHALL display a toast error notification using the existing toastError utility and keep the form open with the user's input preserved

### Requirement 5: Feedback Submission Backend Endpoint with DynamoDB Storage and AI Classification

**User Story:** As a developer, I want a backend endpoint that receives, validates, stores, and automatically classifies user feedback reports, so that the team can efficiently review and act on submitted bugs and feature requests via the admin console.

#### Acceptance Criteria

1. THE Feedback_Submission_Endpoint SHALL accept POST requests at the path /feedback with a JSON body containing: reportType (string, required, "bug" or "feature"), subject (string, required), description (string, required), userEmail (string, required), and userName (string, optional)
2. THE Feedback_Submission_Endpoint SHALL require a valid Cognito JWT via the CognitoAuthorizer, consistent with all other authenticated endpoints in the API
3. WHEN the Feedback_Submission_Endpoint receives a valid request, THE Feedback_Ingestion_Lambda SHALL write a record to the Feedback_Reports_Table containing all submitted fields, the userId extracted from the Cognito JWT, a timestamp (ISO 8601 UTC), a unique reportId (UUID), and the Report_Status set to "active"
4. WHEN the Feedback_Submission_Endpoint receives a valid request, THE Feedback_Ingestion_Lambda SHALL invoke Claude Haiku via Amazon Bedrock to classify the submission as "bug" or "feature_request" and generate a one-sentence AI_Summary of the report
5. THE Feedback_Ingestion_Lambda SHALL store the AI_Classification and AI_Summary in the Feedback_Reports_Table record alongside the user-submitted fields
6. IF the Bedrock invocation fails or times out, THEN THE Feedback_Ingestion_Lambda SHALL store the record with AI_Classification set to "unclassified" and AI_Summary set to an empty string, and SHALL still return HTTP 200 to the user
7. WHEN the Feedback_Submission_Endpoint receives a valid request, THE Feedback_Ingestion_Lambda SHALL also write a structured JSON log entry to CloudWatch Logs containing all submitted fields, the userId, timestamp, and source set to "feedback"
8. WHEN the Feedback_Submission_Endpoint receives a request missing required fields (reportType, subject, or description), THE Feedback_Ingestion_Lambda SHALL return HTTP 400 with a JSON body containing a descriptive validation error
9. IF the Feedback_Submission_Endpoint receives a reportType value other than "bug" or "feature", THEN THE Feedback_Ingestion_Lambda SHALL return HTTP 400 with a JSON body indicating the invalid report type
10. THE Feedback_Ingestion_Lambda SHALL return HTTP 200 with a JSON body containing {status: "submitted"} for every successfully processed feedback report
11. THE Feedback_Ingestion_Lambda SHALL use the shared cors.py utility for CORS headers, the shared structured_logger for logging, and the shared responses module for error responses, consistent with the errorIngestion Lambda pattern
12. IF the Feedback_Submission_Endpoint receives a request body larger than 20 KB, THEN THE Feedback_Ingestion_Lambda SHALL return HTTP 400 with a JSON body indicating the request is too large
13. IF the Feedback_Submission_Endpoint receives a description longer than 5000 characters, THEN THE Feedback_Ingestion_Lambda SHALL truncate the description to 5000 characters and append "[truncated]" before storing

### Requirement 6: Accessibility and Responsive Design

**User Story:** As a user with accessibility needs, I want the feedback modal and menu item to be fully accessible with keyboard navigation, screen reader support, and appropriate touch targets, so that I can use the feature regardless of how I interact with the application.

#### Acceptance Criteria

1. THE Feedback_Menu_Item SHALL have a minimum touch target size of 44x44 pixels (min-h-[44px]), consistent with all other UserMenu items
2. THE Feedback_Modal SHALL include proper ARIA attributes: role="dialog", aria-modal="true", aria-labelledby referencing the DialogTitle, and aria-describedby referencing the DialogDescription
3. THE Feedback_Form SHALL associate each input field with a visible label element using htmlFor/id attributes, so that screen readers announce the field purpose when focused
4. WHEN a validation error is displayed, THE Feedback_Form SHALL associate the error message with the corresponding input using aria-describedby and set aria-invalid="true" on the input, so that screen readers announce the error
5. THE Feedback_Form SHALL support full keyboard navigation: Tab to move between fields, Enter to submit, and Escape to close the modal
6. THE Feedback_Modal SHALL render correctly on viewports from 320px width (small mobile) to 1920px width (large desktop) without horizontal scrolling or content overflow

### Requirement 7: Infrastructure and Deployment

**User Story:** As a developer, I want the feedback infrastructure defined in the SAM template following existing patterns, so that it deploys automatically with the rest of the backend infrastructure.

#### Acceptance Criteria

1. THE Feedback_Ingestion_Lambda SHALL be defined in template.yml with the CognitoAuthorizer, the SharedUtilsLayer (cors, responses, structured_logger), and IAM permissions for CloudWatch Logs write access, DynamoDB read/write access to the Feedback_Reports_Table, and Bedrock InvokeModel access for the Claude Haiku model
2. THE Feedback_Ingestion_Lambda SHALL use the same Python 3.12 runtime, CORS configuration, and environment variables (ALLOWED_ORIGIN) as all other Lambda functions in the project
3. THE Feedback_Ingestion_Lambda SHALL include `import os` at the top of app.py and use `os.environ.get('ALLOWED_ORIGIN', 'https://www.soulreel.net')` for CORS headers, consistent with the CORS rules for this project
4. THE Feedback_Submission_Endpoint path (/feedback) SHALL be added to the API_CONFIG.ENDPOINTS object in FrontEndCode/src/config/api.ts as SUBMIT_FEEDBACK
5. THE Feedback_Ingestion_Lambda SHALL handle OPTIONS preflight requests and return appropriate CORS headers, consistent with the errorIngestion Lambda pattern
6. THE Feedback_Ingestion_Lambda SHALL complete processing (including Bedrock invocation) within 15 seconds to stay within Lambda timeout limits while allowing time for AI classification
7. THE Feedback_Reports_Table SHALL be defined in template.yml as an AWS::DynamoDB::Table resource with PAY_PER_REQUEST billing mode, a partition key of reportId (String), a GSI on submittedAt for date-sorted queries, and PointInTimeRecoveryEnabled set to true
8. THE Feedback_Reports_Table name SHALL be passed to all feedback-related Lambda functions via the TABLE_FEEDBACK_REPORTS environment variable in the SAM Globals
9. THE Feedback_Admin_Lambda SHALL be defined in template.yml with the CognitoAuthorizer, the SharedUtilsLayer, and IAM permissions for DynamoDB read/write/delete access to the Feedback_Reports_Table

### Requirement 8: Admin Feedback Page in Admin Console

**User Story:** As an admin, I want a dedicated "Bugs and Feature Requests" section in the admin console, so that I can review all user-submitted feedback in one place without relying on email or external tools.

#### Acceptance Criteria

1. THE AdminLayout SHALL include a new navigation section labeled "FEEDBACK" containing a single item "Bugs & Requests" with a lucide-react icon (MessageSquare or Bug), linking to /admin/feedback
2. THE Admin_Feedback_Page SHALL be accessible only to authenticated users in the SoulReelAdmins Cognito group, consistent with all other admin pages via the AdminGate component
3. THE Admin_Feedback_Page SHALL display a page title of "Bugs and Feature Requests" prominently at the top
4. THE Admin_Feedback_Page route (/admin/feedback) SHALL be added to the admin routes in App.tsx, nested under the existing AdminGate and AdminLayout route structure

### Requirement 9: Feedback Reports Sortable Table

**User Story:** As an admin, I want to see all feedback reports in a sortable table with key information at a glance, so that I can quickly triage and prioritize bugs and feature requests.

#### Acceptance Criteria

1. THE Admin_Feedback_Page SHALL display a table with the following columns: AI_Classification (bug/feature), AI_Summary, Date Submitted, Report_Status (active/archived), User Name, and User Email
2. THE table SHALL display the Date Submitted column prominently, formatted in a human-readable format (e.g., "Jan 15, 2026 at 3:42 PM")
3. THE table SHALL be sortable by any column when the user clicks the column header, toggling between ascending and descending order
4. THE table SHALL default to sorting by Date Submitted in descending order (newest first)
5. THE table SHALL display the AI_Classification as a colored badge: "Bug" in red/orange and "Feature" in blue/purple, for quick visual distinction
6. THE table SHALL display the Report_Status as a badge: "Active" in green and "Archived" in gray
7. WHEN the Feedback_Admin_Endpoint returns an empty result set, THE Admin_Feedback_Page SHALL display a friendly empty state message (e.g., "No feedback reports yet.")
8. WHILE the Admin_Feedback_Page is loading data from the Feedback_Admin_Endpoint, THE table SHALL display a loading skeleton or spinner

### Requirement 10: Feedback Report Actions (Archive and Delete)

**User Story:** As an admin, I want to archive or delete feedback reports directly from the table, so that I can manage the feedback queue and keep it organized.

#### Acceptance Criteria

1. THE table SHALL include an actions column with an "Archive" button and a "Delete" button for each row
2. WHEN the admin clicks the "Archive" button on an active report, THE Admin_Feedback_Page SHALL send a PATCH request to the Feedback_Admin_Endpoint to update the Report_Status to "archived"
3. WHEN the admin clicks the "Archive" button on an archived report, THE Admin_Feedback_Page SHALL send a PATCH request to the Feedback_Admin_Endpoint to update the Report_Status back to "active" (unarchive)
4. WHEN the admin clicks the "Delete" button, THE Admin_Feedback_Page SHALL display a confirmation dialog before sending a DELETE request to the Feedback_Admin_Endpoint
5. WHEN a report is successfully archived or unarchived, THE Admin_Feedback_Page SHALL update the table row in place without a full page reload
6. WHEN a report is successfully deleted, THE Admin_Feedback_Page SHALL remove the row from the table without a full page reload
7. IF an archive or delete request fails, THEN THE Admin_Feedback_Page SHALL display a toast error notification using the existing toastError utility

### Requirement 11: Feedback Report Detail View

**User Story:** As an admin, I want to click on a feedback report row to see the full details of the submission, so that I can understand the complete context of the bug or feature request.

#### Acceptance Criteria

1. WHEN the admin clicks on a table row (excluding the action buttons), THE Admin_Feedback_Page SHALL open a detail view displaying all submitted fields: Report_Type (user-selected), subject, description, user email, user name, Date Submitted, AI_Classification, and AI_Summary
2. THE detail view SHALL be rendered as a shadcn/ui Dialog or Sheet component, consistent with the application's modal patterns
3. THE detail view SHALL display the Date Submitted prominently at the top of the detail content
4. THE detail view SHALL display the full description text with preserved line breaks and whitespace
5. THE detail view SHALL include a close button and support closing via the Escape key
6. THE detail view SHALL display both the user-selected Report_Type and the AI_Classification side by side, so the admin can compare the user's categorization with the AI's classification

### Requirement 12: Admin Feedback Backend Endpoints

**User Story:** As a developer, I want backend endpoints for listing, retrieving, archiving, and deleting feedback reports, so that the admin console can manage feedback data stored in DynamoDB.

#### Acceptance Criteria

1. THE Feedback_Admin_Endpoint SHALL expose GET /admin/feedback to list all feedback reports from the Feedback_Reports_Table, sorted by submittedAt descending by default
2. THE Feedback_Admin_Endpoint SHALL expose GET /admin/feedback/{reportId} to retrieve a single feedback report by its reportId
3. THE Feedback_Admin_Endpoint SHALL expose PATCH /admin/feedback/{reportId} to update the Report_Status of a feedback report (accepting a JSON body with {status: "active"} or {status: "archived"})
4. THE Feedback_Admin_Endpoint SHALL expose DELETE /admin/feedback/{reportId} to permanently delete a feedback report from the Feedback_Reports_Table
5. THE Feedback_Admin_Endpoint SHALL require a valid Cognito JWT via the CognitoAuthorizer, consistent with all other authenticated endpoints
6. IF the Feedback_Admin_Endpoint receives a request for a reportId that does not exist, THEN THE Feedback_Admin_Lambda SHALL return HTTP 404 with a JSON body containing a descriptive error
7. THE Feedback_Admin_Lambda SHALL use the shared cors.py utility for CORS headers, the shared structured_logger for logging, and the shared responses module for error responses
8. THE Feedback_Admin_Endpoint paths (/admin/feedback, /admin/feedback/{reportId}) SHALL be added to the API_CONFIG.ENDPOINTS object in FrontEndCode/src/config/api.ts as ADMIN_FEEDBACK_LIST, ADMIN_FEEDBACK_DETAIL, ADMIN_FEEDBACK_UPDATE, and ADMIN_FEEDBACK_DELETE
